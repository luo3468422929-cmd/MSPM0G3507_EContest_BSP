# 代码阅读顺序与注释规范

这份文档给第一次接触本工程的队员使用。不要从 `Bsp` 的 DriverLib 细节开始硬啃，先看程序怎样跑，再追到具体器件。

## 一、推荐阅读顺序

```text
User/main.c
  ↓ 反复调用 Task_Run()
User/task.c
  ↓ 正常模式调用整车控制；测试模式调用 Test_Run()
Control/car_control.c
  ↓ 读取循迹和编码器，计算 PID，再设置电机
Hardware/track.c / encoder.c / motor.c
  ↓ 调用 MCU 总线和 GPIO/PWM 通道
Bsp/i2c.c / uart.c / timer.c / board_pins.h
  ↓
TI DriverLib + SysConfig 生成配置
```

建议第一遍只阅读每个文件顶部的 `@brief`、头文件公共接口和核心流程注释，不需要立刻理解每条 DriverLib API。

## 二、四层分别负责什么

### User：题目流程和现场参数

- `main.c`：只初始化并永久轮询；通常不改。
- `user_config.h`：速度、PID、方向、机械参数、测试选择和功能开关。
- `task.c`：上电、等待、启动、运行、停车、终点等题目状态机。
- `test.c`：每个模块的独立上板测试。

比赛改题目流程优先改这一层，但物理引脚不写在这里。

### Control：算法和多模块协调

- `pid.c`：通用位置式/增量式 PID。
- `car_control.c`：循迹误差变成左右轮目标 RPM，再运行双速度环。
- `scheduler.c`：决定 5/10/20/100 ms 任务本轮是否到期。

这一层不应包含 `DL_GPIO_*`、PA17 等硬件细节。

### Hardware：具体器件

- `motor.c` 知道 TB6612 的方向、STBY 和 PWM 语义。
- `encoder.c` 知道 A/B 两相 x2 解码和 RPM 计算。
- `track.c` 知道八路模块地址、寄存器和位顺序。
- `imu.c` / `imu_protocol.c` 知道 NCU 命令和 0x5A 数据帧。
- `lcd.c` 知道 ST7735S 命令、字库和 RGB565。

更换同类器件时优先替换 Hardware 实现，并尽量保持头文件接口不变。

### Bsp：MSPM0G3507 基础通道

- `board_pins.h`：SysConfig 逻辑名称映射。
- `uart/i2c/spi/timer`：DriverLib 通道和有限超时。
- `ring_buffer`：ISR 与主循环之间的字节缓冲。
- `board.c`：整板初始化顺序和急停汇总。

物理引脚、外设实例和复用只在 `empty.syscfg` 修改；不要直接编辑 `Debug/syscfg` 生成文件。

## 三、如何追一条调用链

以“循迹后驱动电机”为例：

1. `main()` 反复调用 `Task_Run()`；
2. 10 ms 到期且状态为 `TASK_RUNNING` 时调用 `CarControl_Update(nowMs)`；
3. `CarControl_Update()` 先更新编码器，再调用 `Track_Update()`；
4. 转向 PID 把 `positionError` 变成左右目标 RPM 差；
5. 左右速度 PID 根据实际 RPM 计算占空比；
6. `Motor_SetDutyPair()` 保存目标，`Motor_RampUpdate()` 平滑写入 TB6612；
7. 最终 GPIO/PWM 实例来自 `board_pins.h` 和 `empty.syscfg`。

排查问题时沿这条链逐层看数据，不要一开始同时改 PID、方向和引脚。

## 四、比赛增加新模块的标准步骤

1. 在 `empty.syscfg` 配置 GPIO/UART/I2C/SPI/ADC 和逻辑名称；
2. 在 `Bsp/board_pins.h` 映射生成名称；
3. 在 `Hardware` 新增器件 `.h/.c`，公共函数使用现有命名风格；
4. 在 `User/test.h/.c` 增加独立测试，先验证接线和协议；
5. 在 `Control` 或 `User/task.c` 接入正式流程；
6. 运行 `Tests/Build/verify_build.ps1`，再按模块流程上板。

若以后更换循迹模块，应继续提供 `Track_Init()`、`Track_Update()`、`Track_GetData()`，这样控制层不用跟着协议变化。

## 五、项目注释标准

每个正式源码文件开头包含：

```c
/**
 * @file module.c
 * @brief 一句话说明职责。
 *
 * 所属层：说明调用边界。
 * 主要依赖：说明依赖的下层模块。
 * 比赛修改：说明现场通常修改哪里。
 */
```

公共接口至少写清功能、参数单位、返回值和初始化/调用周期要求。源文件重点解释状态机、硬件顺序、中断共享数据和失败后的安全行为。

不要写这种无意义注释：

```c
count++; /* count 加一 */
```

应该解释为什么加、谁读写、溢出是否允许。代码变化后必须同步修改注释，错误注释比没有注释更危险。

## 六、看到这些注释必须重新上板验证

- 引脚、电平有效极性、内部/外部上拉；
- TB6612 STBY、方向和刹车语义；
- 编码器方向、每圈计数、减速比和轮径；
- I2C 地址、寄存器、位顺序和上拉电压；
- UART 波特率、帧格式和 TX/RX 交叉；
- LCD 控制器、MADCTL、偏移和 SPI 模式；
- PID 调用周期、输出限幅和电机启动死区。

文档和自动编译只能证明软件结构一致，不能替代万用表、架空车轮测试和实车低速验证。
