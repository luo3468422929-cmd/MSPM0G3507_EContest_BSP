# 正式源码中文注释补全设计

**状态：已确认，等待实施计划。**

## 目标

为 `Bsp/`、`Hardware/`、`Control/`、`User/` 下全部 60 个正式 `.c/.h` 文件建立统一、适合电赛新手阅读的中文注释。注释帮助队员快速理解模块职责、调用顺序、配置入口和硬件注意事项，但不逐行复述显而易见的 C 语句。

本次只修改注释和说明文档，不改变接口、控制参数、引脚、时序、状态机或硬件行为。

## 注释标准

### 文件级

每个正式源码文件都要说明：

- 文件负责什么；
- 属于 `Bsp`、`Hardware`、`Control` 或 `User` 哪一层；
- 依赖哪些下层模块；
- 比赛换引脚、换器件或改策略时应该改哪里。

### 头文件

- 枚举：解释每个状态或编号的含义；
- 结构体：解释关键字段、单位和数据有效条件；
- 配置宏：说明取值范围、单位、极性及修改影响；
- 公共函数：说明功能、入口参数、返回值和使用注意事项；
- 中断调用接口：明确由哪个 IRQ 调用、不能执行哪些阻塞操作。

简单的 include guard、显然的常量和无歧义字段不堆砌重复注释。

### 源文件

- 静态变量：说明保存的状态及由中断还是主循环访问；
- 初始化：说明硬件安全顺序和失败处理；
- 核心流程：按数据流或状态机分段说明“为什么这样做”；
- 中断：说明触发源、清标志顺序、共享数据和实时性约束；
- 错误路径：说明超时、通信失败、丢线或急停后的安全行为；
- 算法：说明输入、输出、单位、限幅和关键公式，不写无关理论推导。

## 分层重点

- `Bsp`：解释 SysConfig 逻辑实例、DriverLib 调用边界、超时和 ISR/主循环通信；
- `Hardware`：解释器件协议、接线假设、方向/极性、采样和状态转换；
- `Control`：解释 PID、调度、循迹和双轮控制的数据流及限幅；
- `User`：解释上电流程、测试选择、ARMING/RUNNING/STOPPED 状态和比赛改动入口。

## 新手阅读文档

新增 `docs/代码阅读顺序与注释规范.md`，内容包括：

1. 推荐阅读顺序：`main` → `task` → `car_control` → `Hardware` → `Bsp`；
2. 四层分别能改什么、不能改什么；
3. 如何从一个公共接口追到具体硬件实现；
4. 比赛新增模块时如何保持同样的注释风格；
5. 哪些注释属于硬件约束，修改前必须重新上板验证。

## 验收标准

- 60 个正式 `.c/.h` 文件均具有文件级中文说明；
- 所有公共接口、关键结构体、状态机、ISR 和核心数据流均有有效注释；
- 不为普通赋值、括号或一眼可懂的判断添加逐行废话；
- 不引入 `TODO`、`TBD` 或与现有配置矛盾的描述；
- `git diff` 除注释和文档外不应出现可执行代码变化；
- `Tests/Build/verify_build.ps1`、Host 断言、SysConfig 和 TI Clang 完整构建全部通过。

## 提交策略

设计说明单独提交。实施时按 `Bsp`、`Hardware`、`Control`、`User + docs` 分批检查，最终以一个注释补全提交收尾，便于需要时独立回退且不影响已经验证的功能提交。
