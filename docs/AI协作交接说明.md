====================
三、项目分层约定
====================

Bsp：
负责 MSPM0G3507 的底层 DriverLib 封装，例如：

- UART
- I2C
- SPI
- Timer
- RingBuffer
- Board 初始化
- SysConfig 引脚映射

Hardware：
负责具体外设和器件驱动，例如：

- TB6612 电机
- 编码器
- 亚博八路循迹模块
- ST7735S LCD
- NCU 惯导串口接收
- 按键
- LED
- 滤波器

Control：
负责算法和模块协调，例如：

- PID
- 双轮速度闭环
- 循迹控制
- 调度器

User：
负责项目流程，例如：

- main
- Task_Run
- 测试模式
- 正常跑车状态机
- 比赛参数配置
- 启停和急停逻辑

不要把硬件寄存器、GPIO 操作、物理引脚写进 Control 或 User 层。

====================
四、硬件和配置约束
====================

主控：

- MSPM0G3507
- LQFP-48
- TI DriverLib
- SysConfig
- CCS 工程

配置规则：

- 物理引脚和外设实例优先修改 empty.syscfg；
- Bsp/board_pins.h 只做 SysConfig 名称映射；
- 不要直接修改 Debug/syscfg 下的生成文件；
- Debug/ 是构建产物，不要把它当作源码修改；
- 修改引脚前必须检查是否和现有外设冲突；
- 不要凭记忆填写引脚，必须读取当前 empty.syscfg 和 board_pins.h。

LCD：

- 当前为 ST7735S SPI LCD；
- BL 背光直接连接 3.3 V；
- PA24 不负责背光；
- 不要重新把 PA24 当作 LCD 背光控制脚；
- LCD 的 SPI、CS、DC、RES 引脚以当前 SysConfig 为准。

循迹：

- 当前使用亚博八路带 MCU 灰度模块；
- 通过 I2C 通信；
- 黑白校准后才能正常获得有效数据；
- 不要把它当成普通 GPIO ADC 五路模块；
- 地址、寄存器、位顺序和校准行为必须阅读 Hardware/track.c、track.h 以及 docs 后再判断；
- 以后更换 12 路模块时，不要直接假设协议相同。

I2C：

- PA0/PA1 尽量不要使用；
- 如果必须使用，需要确认外部上拉电阻；
- SDA/SCL 上拉到 3.3 V，不要上拉到 5 V；
- I2C 错误时要考虑模块供电、共地、上拉、电平和地址，而不是只改软件。

电机：

- 使用 TB6612；
- 必须共地；
- 电机电源和 MCU 逻辑电源要区分；
- STBY、AIN1/AIN2、BIN1/BIN2、PWMA/PWMB 的逻辑必须以当前代码和 SysConfig 为准；
- 测试电机时先架空车轮；
- 现场不能为了测试随便取消急停；
- 修改方向前先确认是软件方向配置、左右接线还是电机机械安装方向。

编码器：

- 当前为 GPIO A/B 两相正交解码；
- 使用 x2 计数；
- A、B 四路 GPIO 上升沿中断；
- 速度使用固定时间窗和滤波；
- 速度计算应使用真实 dt；
- 方向通过配置宏修正，不要随意改公共解码逻辑；
- 具体引脚、每圈计数、减速比、轮径必须读取当前 user_config.h 和 encoder 文件；
- 不要仅凭电机型号猜 RPM 或 CPR。

NCU 惯导：

- 通过 UART 接收；
- 接收层、协议解析层和应用显示层分离；
- UART 中断只负责收字节；
- 协议解析在主循环执行；
- 先确认串口号、波特率、帧头、长度、校验和，再判断数据是否异常；
- 不要把串口乱码直接判断为惯导损坏。

====================
五、代码修改规则
====================

如果我只是问“这是什么”“为什么”，只解释，不要改代码。

如果我说“帮我修改”“你直接改”“实现这个功能”，再开始修改。

修改前必须：

1. 检查 git status；
2. 阅读相关源码和配置；
3. 说明准备修改哪些文件；
4. 说明是否会改变硬件行为；
5. 如果是危险或大范围修改，先建议创建回退 commit；
6. 不修改无关文件；
7. 不修改 Debug/syscfg 生成文件；
8. 使用 apply_patch 修改源码；
9. 修改完成后运行验证。

不要为了让编译通过而删除功能、屏蔽报警、绕过急停或修改测试规则。

如果遇到不确定的硬件问题：

- 不要自以为是；
- 明确列出软件、接线、供电、模块损坏、测量方式等可能性；
- 优先给我一个最小风险的排查步骤；
- 只有涉及不可逆硬件修改、重新焊线或修改 PCB 时才必须先问我；
- 普通软件检查可以直接继续。

- 编码器中断；
- 电机方向和缓升；
- PID 测试模式；
- 八路循迹 I2C；
- LCD 接口；
- 惯导显示；
- 用户层 API；
- Host 测试；
- ARM GCC 编译；
- SysConfig；
- TI Clang 编译和链接。

如果验证失败：

1. 先阅读完整错误；
2. 定位根因；
3. 不要直接猜测修复；
4. 修复后重新运行同一条命令；
5. 告诉我哪些检查通过、哪些失败。

如果只是文档或注释修改，也至少运行：

git diff --check

如果修改了源码，必须尽量运行完整 verify_build.ps1。

====================
七、标准调试流程
====================

遇到硬件现象时，按以下顺序排查：

1. 现象是否可以稳定复现；
2. 当前运行的是哪个 TEST 模式；
3. 当前 STARTUP_TEST 配置是什么；
4. 电源电压是否正确；
5. MCU 和外设是否共地；
6. 线缆、杜邦线、插接方向是否可靠；
7. SysConfig 引脚是否与实际接线一致；
8. 串口是否选择正确；
9. 是否有 I2C 地址或校准问题；
10. 是否有编码器信号浮空；
11. 是否有电机方向和 STBY 问题；
12. 最后才修改算法或驱动代码。

不要一看到现象就直接改 PID、改方向宏或改协议。
